<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Attack panel -> AttackSimulator (XSS) | XSS payloads contain script tags -- these are sent as HTTP request parameters, not rendered in the attack panel UI |
| Attack panel -> AttackSimulator (CC Flood) | CC flood sends 4000 requests (20 threads x 200) -- could impact local Docker performance |
| Attack panel -> AttackSimulator (Brute Force) | Simulated login attempts to /login endpoint with fake credentials -- no real authentication is bypassed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07 | Denial of Service | CC Flood attack (ATTACK_CONFIG cc_flood: 20 threads, 200 req/thread, 0.001s delay) | mitigate | CC flood attacks localhost nginx only. The st.session_state.attack_running gate (Plan 01-01 T-01-03) prevents concurrent attacks including concurrent CC floods. If Docker nginx becomes unresponsive, it recovers on container restart. Total requests are bounded at 4000 per launch. |
| T-01-08 | Information Disclosure | Attack history panel | mitigate | Attack history is stored in Streamlit session state (in-memory, per-process). History is cleared on app restart. History shows only attack metadata (type, IP, counts), not raw payloads. The "清除历史" button allows instant clearing. |
| T-01-09 | Tampering | Attack type selector | mitigate | Attack type values are validated against the attack_launchers dict keys before dispatch. Unknown attack types produce a user-visible error via st.error() rather than crashing. The user cannot inject arbitrary Python code through the selector. |

## Cross-Plan Threat Inheritance

Threats T-01-01 through T-01-06 and T-01-SC from Plan 01-01 remain applicable and are not repeated here. The mitigations in Plan 01-01 cover the extended attack panel as well.
</threat_model>
